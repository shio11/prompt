# Pythonコード生成ルール

Pythonコードを実装・生成する際は、以下の設計要件・構成ルールに常に従うこと。

## 設計要件
- オブジェクト指向設計を用い、責務ごとにクラスを分割すること
- 各クラスは単一責任の原則（SRP）に従うこと
- インスタンス変数はカプセル化し、外部から直接アクセスさせないこと
  - 非公開属性には `_name`（内部利用）または `__name`（名前マングリングが必要な場合）の命名規則を使う
  - 外部への公開が必要な属性は `@property` / `@<attr>.setter` を介してアクセスさせる
  - setter側でバリデーション（型チェック・値域チェック等）を行い、不正な状態のオブジェクトが生成されないようにする
- クラスの公開インターフェース（メソッド・プロパティ）と内部実装を明確に分離すること
- 継承よりコンポジションを優先し、継承は「is-a」関係が明確な場合のみ使用すること
- 型ヒント（`typing` モジュール含む）を全ての引数・戻り値・属性に付与すること
- 不変にできる属性は `@dataclass(frozen=True)` や `__slots__` の活用も検討すること
- マジックメソッド（`__repr__`, `__eq__` など）を適切に実装し、デバッグ・比較を容易にすること

## プログラム構成

まずプロジェクトの規模を判定し、該当する構成ルールに従うこと。

### 規模の判定
- Webアプリケーション、データベースを利用するプロジェクトなど、中〜大規模なプロジェクトの場合 → 「中〜大規模プロジェクト向け構成」に従う
- 単一目的のスクリプト・ツールなど、上記に該当しない小規模なプロジェクトの場合 → 「小規模プロジェクト向け構成」に従う

### 小規模プロジェクト向け構成
- ディレクトリ分割はせず、`models.py` / `services.py` / `main.py` の3ファイル構成とすること
  - `models.py`: 値オブジェクト・DTO・Enum等のデータ構造をまとめる
  - `services.py`: ビジネスロジック・処理を担うクラスをまとめる
  - `main.py`: エントリーポイント（下記「共通ルール」参照）
- 各ファイルが200〜300行程度を超えて肥大化した場合や責務が明確に異なるクラス群が混在してきた場合は、その時点で中〜大規模プロジェクト向け構成（ディレクトリ分割）への移行を検討すること

### 中〜大規模プロジェクト向け構成（Webアプリケーション・DB利用等）
- 責務のレイヤー（例: `models/`, `services/`, `repositories/`）ごとにディレクトリを分割すること
- ファイル分割は「クラス単位」ではなく「責務のまとまり単位」で行うこと
  - 密接に関連する小規模なクラス（値オブジェクト、DTO、Enum等）は同一ファイルにまとめてよい
  - 1ファイルの目安は200〜300行程度を上限とし、これを超える場合や責務が明確に異なる場合にファイルを分割すること
  - 独立して再利用される可能性が高いクラス・複雑度の高いクラスは個別ファイルに分離すること

### 共通ルール（規模によらず）
- `main.py` はエントリーポイントとし、以下の役割のみを持たせること
  - 各サブモジュールのインポート
  - 必要なオブジェクトの生成（依存性の組み立て）
  - 各サブプログラムの呼び出し・処理フローの制御
  - `main.py` 自体にはビジネスロジックやクラスの内部実装を書かないこと
- `main.py` は `if __name__ == "__main__":` ブロックから `main()` 関数を呼び出す形にすること
- 各サブモジュールは `main.py` からimportされることを前提に、単体でも再利用・テスト可能な設計にすること
- ディレクトリ構成案を最初に提示してから実装すること

## 出力形式
新規にPythonプログラムを実装する際は、以下の順で提示すること。
- ディレクトリ構成（ツリー形式）
- 各ファイルの実装コード
- クラス設計の意図（なぜそのカプセル化・分割にしたか）を簡潔に説明
